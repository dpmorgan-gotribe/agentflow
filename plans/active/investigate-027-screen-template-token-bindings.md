# investigate-027: screen template token-binding consistency + design-intent revision surface

## Empirical motivator

bug-098 (closed 2026-05-14 by diagnostic) walked through 4 hypotheses about why parity-verify's computed-styles audit didn't surface a user-reported "tabs grey vs light-blue" finding. All 4 ruled out. The actual root cause:

- `docs/screens/webapp/books-list.html` `.seg` class: hardcodes `background: #f1f4f5` (light grey)
- `docs/mockups/style-1/webapp/books-list.html` `.seg` class: same hardcoded `#f1f4f5`
- Build's kit `Tabs` primitive: renders to the same `#f1f4f5` via its kit-variant CSS

Both source-of-truth files + the build itself produce identical grey. No drift exists between them. The audit correctly finds nothing. Yet the user expected light-blue → the design-intent diverges from the source-of-truth without a revision surface.

The pipeline correctly preserves whatever the screen template says. But the screen template's relationship to "design intent" is opaque — there's no token-name vocabulary in the templates, just hex values.

## Questions to answer

1. **Where do screen-template hex values originate?** Does `/screens` (the skill) generate them from `selected-style.json`? From the brief? From defaults baked into the screens skill's prompt?
2. **Do screen templates ever reference NAMED tokens?** A grep across 5+ shipped projects' `docs/screens/**/*.html` — quantify the ratio of hardcoded hex vs `var(--token-name)`. If it's 100% hardcoded, that's the gap.
3. **What's the revision flow when a user sees a divergence?** Today: file a bug? Update screens.html manually? Re-pick a style at gate 2? Update `selected-style.json` and regenerate? None of these are documented as canonical.
4. **Would a `audit-design-tokens-vs-screens` verifier step be tractable?** It'd compare screen-rendered colors against the named tokens in `selected-style.json`. Output: which screen elements use UN-token-bound colors. Would catch the gap. Cost: requires screen templates to carry token-name annotations OR the audit to do reverse-lookup from hex to nearest token.
5. **Could the screens skill emit hardcoded hex + a `data-token` annotation?** E.g. `<div class="seg" data-token-background="accent-50" style="background: #e6f3ff">`. The audit reads `data-token-background`, looks up the current token value in `selected-style.json`, compares against the computed background-color. This is non-breaking + retro-applicable.
