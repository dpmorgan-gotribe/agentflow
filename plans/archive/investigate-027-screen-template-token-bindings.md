# investigate-027: screen template token-binding consistency + design-intent revision surface

## Empirical motivator

bug-098 (closed 2026-05-14 by diagnostic) walked through 4 hypotheses about why parity-verify's computed-styles audit didn't surface a user-reported "tabs grey vs light-blue" finding. All 4 ruled out. The actual root cause:

- `docs/screens/webapp/books-list.html` `.seg` class: hardcodes `background: #f1f4f5` (light grey)
- `docs/mockups/style-1/webapp/books-list.html` `.seg` class: same hardcoded `#f1f4f5`
- Build's kit `Tabs` primitive: renders to the same `#f1f4f5` via its kit-variant CSS

Both source-of-truth files + the build itself produce identical grey. No drift exists between them. The audit correctly finds nothing. Yet the user expected light-blue → the design-intent diverges from the source-of-truth without a revision surface.

The pipeline correctly preserves whatever the screen template says. But the screen template's relationship to "design intent" is opaque — there's no token-name vocabulary in the templates, just hex values.

## Questions to answer

1. **Where do screen-template hex values originate?** Does `/screens` (the skill) generate them from `selected-style.json`? From the brief? From defaults baked into the screens skill's prompt?
2. **Do screen templates ever reference NAMED tokens?** A grep across 5+ shipped projects' `docs/screens/**/*.html` — quantify the ratio of hardcoded hex vs `var(--token-name)`. If it's 100% hardcoded, that's the gap.
3. **What's the revision flow when a user sees a divergence?** Today: file a bug? Update screens.html manually? Re-pick a style at gate 2? Update `selected-style.json` and regenerate? None of these are documented as canonical.
4. **Would a `audit-design-tokens-vs-screens` verifier step be tractable?** It'd compare screen-rendered colors against the named tokens in `selected-style.json`. Output: which screen elements use UN-token-bound colors. Would catch the gap. Cost: requires screen templates to carry token-name annotations OR the audit to do reverse-lookup from hex to nearest token.
5. **Could the screens skill emit hardcoded hex + a `data-token` annotation?** E.g. `<div class="seg" data-token-background="accent-50" style="background: #e6f3ff">`. The audit reads `data-token-background`, looks up the current token value in `selected-style.json`, compares against the computed background-color. This is non-breaking + retro-applicable.

## Attempt Log

### 2026-05-14 — investigation complete; recommendation: dual-path (A + E)

**Methodology** ran ~30 min within the 60-min time-box. Findings:

#### Finding 1 — screen templates are 100% hardcoded hex, 0% named tokens (empirically confirmed across 4 projects)

Grep across 4 shipped projects (reading-log-01, reading-log-02, reading-log-pre-bugs, reading-log-pre-build):

| Project                          | `var(--` count | hex count |
| -------------------------------- | -------------: | --------: |
| reading-log-01 books-list        |              0 |        72 |
| reading-log-02 books-list        |              0 |        72 |
| reading-log-pre-bugs books-list  |              0 |        72 |
| reading-log-pre-build books-list |              0 |        72 |

Universal pattern. Screens never reference tokens.

#### Finding 2 — the canonical tokens DO exist as `packages/ui-kit/src/tokens/tokens.json` (W3C DTCG)

reading-log-02's kit has:

- `tokens.json` — W3C DTCG-format design tokens (machine-readable)
- `tokens.css` — generated CSS custom properties (`--color-accent-500: #4a7c7e`, full neutral / accent / warm / surface / status / semantic ramps)
- The kit's components use `var(--color-...)` internally; rendered builds get correct token resolution

The break is: `/screens` (skill) generates `docs/screens/webapp/*.html` with raw hex literals copied from style markdown — not `var(--color-...)` references. The token NAME information is lost at the screen-emit boundary.

#### Finding 3 — `docs/selected-style.json` is a 14-key pointer (no enumerated tokens)

```json
{
  "version": "1.0",
  "styleId": "style-1",
  "styleName": "Quiet Modern",
  "dials": { "design_variance": 3, ... },
  "iconLibrary": "phosphor",
  "mockupsManifest": "docs/mockups/style-1/manifest.json",
  ...
}
```

The actual token VALUES live in `packages/ui-kit/src/tokens/tokens.json` (downstream of the style pick). `selected-style.json` records WHICH style was picked + dial values, NOT the resolved token values.

#### Finding 4 — `/screens` skill's emission path bridges design → screen but drops token names

The screens skill (`.claude/skills/screens/SKILL.md`) reads `docs/selected-style.json` + `docs/analysis/{platform}/screens.json` + kit's components-plan + style.md. It composes HTML with kit-component data attributes. But its CSS-emission step inlines hex literals from the style markdown, severing the link to `tokens.json`.

#### Finding 5 — the user's "sidebar not full height" bug is a layout-token analog of the same gap

User clarification (2026-05-14): the original "sidebar only 227.333px" complaint meant the sidebar should be FULL viewport height, not the kit's default mid-page height. Same gap shape: kit-default + screen-template + build all agree on the non-full-height value; the user's design-intent (100vh / dvh) diverges from all three. The audit can't catch what isn't a divergence between its source-of-truth pair.

This generalizes: tokens (colors) AND layout primitives (sidebar height, header alignment, page-grid columns) share the same gap class.

### Recommendation: dual-path A + E

**Path A — emit `data-token-*` annotations on screen templates** (structural fix):

Extend `/screens` skill so when a screen-template HTML element gets a hex color OR a layout dimension, it also gets a `data-token-{property}="{token-name}"` annotation referencing the token from `packages/ui-kit/src/tokens/tokens.json`.

Example before/after:

```html
<!-- before -->
<div class="seg" style="background: #f1f4f5">
  <!-- after -->
  <div
    class="seg"
    style="background: #f1f4f5"
    data-token-background="neutral-100"
  ></div>
</div>
```

The audit then:

1. Reads `tokens.json` at audit-time → resolves `neutral-100` → `#F5F4EF`
2. Compares the screen's `data-token-background` resolved value against the BUILD's computed `background-color`
3. Compares against the screen's INLINE hex value (catches drift between template + token)
4. Fires `token-binding-drift` divergences when they differ

Non-breaking — adds annotations, doesn't change existing hex emission. Empirically tractable. Catches the bug-098 class structurally. ~half-day implementation in `/screens` skill + new `scripts/audit-design-tokens-vs-screens.mjs`.

**Path E — stack-skill layout-decision content** (preventive fix; user's instinct):

Codify common layout-decision pitfalls in `.claude/skills/agents/front-end/react-next/SKILL.md` § Layout conventions. Concretely:

```md
## §X — AppShell layout invariants (factory-wide; prevents bug-099-class regressions)

- **AppShell sidebar MUST fill viewport height** (`100vh` / `100dvh` / `h-screen` / `min-h-screen`).
  Default kit behavior may not enforce this; explicit class is non-negotiable for
  apps shipping with AppShell.
- **AppShell topbar MUST span full viewport width** (`w-screen` / `w-full` with
  `<main>` flex layout). Empirical: missing this produces ~227px width and search-bar
  off-center.
- **Brand wordmark in topbar's left slot when mockup includes one** — check the
  screens/{platform}/{screen}.html for a brand element; if present, builder MUST
  emit it.
- **Page-bottom elements (pagination, footer) MUST be inside the AppShellMain
  scrollable area**, not below the viewport.
```

Fast to ship (single file edit). Catches the specific empirical user-found classes (sidebar-not-full-height, topbar-not-full-width, brand-absent) by giving the builder agent explicit invariants to honor. Doesn't catch unknown layout pitfalls — relies on enumeration. Best as a low-leverage immediate fix while Path A's structural fix lands.

**Path B/C/D evaluation**:

- B (audit-design-tokens-vs-screens as a new verifier step) — natural follow-on to Path A; depends on A landing first.
- C (defer) — rejected; the empirical evidence is here right now. Deferring kicks the can on real bugs.
- D (re-frame as user-education) — partial truth (the user CAN edit selected-style.json) but it's not a documented revision path AND the per-tab-color OR per-sidebar-height customizations don't have a clear surface today. Re-framing alone doesn't solve.
